背景问题

vLLM 处理长文本时，HuggingFace 分词器是单线程 CPU 瓶颈。

短文本(数百 token)时分词耗时可忽略，但输入达 100k+ 字符时，单线程分词耗时可达数秒。

HfRenderer._tokenize_prompt_async()

    - queue.put() -> _batch_encode_loop()
      - ThreadPoolExecutor 中的单线程 HF 分词器
        - 整个请求串行化，不跨块并行

算法详解

分块 → 并行分词 → 重叠合并。核心突破在 ca-偏移修复。

算法流程

输入文本 S (N 字符)
|
- 1. 按 Lc 拆分为块 (每块 Lc+LO 字符)
|   Lc = 分块大小(字符), LO = 重叠大小(字符)
|
- 2. 并行分词
|   - 块数 ≤10 → 线程池 (零 IPC)
|   - 块数 ≥20 → 进程池(spawn) (独立 GIL)
|
- 3. 用 offsets 在重叠区找 anchor
|   同时比 token_id + 字符位置
|   ca-偏移双重循环解决 BPE 边界问题
|
- 4. 按 anchor 位置确定每块贡献段
|
- 5. 拼接为完整 token 序列

ca-偏移修复

项目启发自 LoPT 论文 (Huawei, arXiv:2511.04952)：通过重叠切分+并行分词+anchor合并实现无损并行分词。本文在此基础上发现了大词表 tokenizer 的 BPE 边界偏移问题并独立解决。
问题： 原始算法只比 pa[pv+k] == ca[k]，ca 始终从索引 0 开始。BPE 在块边界偏移 1-2 token -> 整段序列错位 -> 全部匹配失败 -> 回退串行。
ca-偏移修复是本文原创发现，不在 LoPT 论文中。 论文假设 BPE 在重叠区内总能稳定对齐（ca[0] 即匹配起点）。但大词表 tokenizer（Qwen3.5 248K、DSV4 Pro 128K）在块边界处不稳定：chunk_{i+1} 无左侧上下文时会多分 1-2 个 token，整段偏移。此修复是本项目独立发现并从"全部 fallback"到"稳定 2x+"的关键突破。
修复前 — ca[0] 对齐失效:

chunk_i overlap (有上下文):   offsets=[(169,175), (175,178), (178,182), (182,185), (185,188)]
                                         pa[0]      pa[1]      pa[2]      pa[3]      pa[4]

chunk_{i+1} overlap (无上下文): offsets=[(170,171), (171,175), (175,178), (178,182), (182,185)]
                                         ca[0]      ca[1]      ca[2]      ca[3]      ca[4]

   pa[0]=(169,175)  vs  ca[0]=(170,171)  → ✖ 不匹配
   pa[1]=(175,178)  vs  ca[1]=(171,175)  → ✖ 不匹配
   pa[2]=(178,182)  vs  ca[2]=(175,178)  → ✖ 不匹配
   全部失败 → fallback → 1.3-1.4x 更慢

原因此 chunk_{i+1} 从字符 170 开始，BPE 将 "b" 和 "egan" 分为两个 token，
而 chunk_i 中有上文，" began" 被合为一个 token (169,175)，导致整段偏移。
修复后 — ca 偏移匹配（尝试所有 pv, qv 起始对）:

chunk_i overlap:   offsets=[(169,175), (175,178), (178,182), (182,185), (185,188)]
                                  pa[0]      pa[1]      pa[2]      pa[3]      pa[4]
                                              ↑ pa[1]=(175,178)

chunk_{i+1} overlap: offsets=[(170,171), (171,175), (175,178), (178,182), (182,185)]
                                  ca[0]      ca[1]      ca[2]      ca[3]      ca[4]
                                                       ↑ ca[2]=(175,178) = pa[1]

   pa[1]=(175,178) == ca[2]=(175,178)  → ✔ 匹配成功！
   继续: pa[2]=(178,182) == ca[3]=(178,182)  → ✔
   继续: pa[3]=(182,185) == ca[4]=(182,185)  → ✔
   anchor = (pa_start=1, ca_start=2, length=3)  → 正确合并 → 0.47-0.55x 加速
修复前 — 全部回退
只对齐 ca[0]
while k < mk and pa[pv+k] == ca[k]:
    k += 1
→ 偏移 1 token 就找不到
→ 回退到串行
→ 1.3-1.4x 更慢
修复后 — 稳定 2x+
尝试 ca 偏移
for qv in range(len(ca)):
    while k < mk and pa[pv+k] == ca[qv+k]:
        k += 1
→ 找到偏移后的匹配
→ 正确合并
→ 0.47-0.55x 加速

语料构造 + 中英文差异

测试数据的构造方式和中英文性能差异的根本原因。

中文：20 段独立新闻短文(~80字/段)，涵盖 12 个主题。重复拼接到目标字符数。

英文：20 段独立英文新闻短文(~60词/段)。英文 chars/token 比约 4:1，远高于中文 1:1。

因素中文英文影响
字符/token 比~1:1~4:1英文每 token 处理 4x 字符，并行收益更大
每 token 串行耗时~2.1us~3.2us英文分词器更慢(更多合并规则)
同 token 下字符数少多英文块数更多 → 并行度更高
重叠大小16 字符32-64 字符英文需要更大重叠
线程池加速~1.35x~1.29x中文线程池略好(合并开销小)
进程池加速~6.2x~8.2x英文明显更快(块多/字符多)
