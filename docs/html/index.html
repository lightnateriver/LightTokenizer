01 · 背景

HF PreTrainedTokenizerFast.encode() 底层 Rust BPE，对一段文本完全串行。>100K tokens 时是不可忽略的 CPU 热点。

Qwen3.5 248K 词表~2.1μs/tok（ARM 核）, 1024K→2.1s vLLM 现状apply_chat_template(tokenize=True) 单线程串行 论文LoPT — Lossless Parallel Tokenization (arXiv:2511.04952)
LoPT 方案简述
文本 S（N 字符） ├── Chunk[0] (Lc+LO) ── tokenize ── ids[0] ├── Chunk[1] (Lc+LO) ── tokenize ── ids[1] ← 并行 ├── ... ▼ 重叠区找最长 token 匹配 → anchor → 拼接完整 ids

变量：Lc(chunk大小), LO(重叠), N_threads(并行度)

02 · 方案演进：ProcessPool → ThreadPool

论文原始方案： 多进程（ProcessPool）——每 chunk 分给子进程 tokenize，结果 IPC 传回。

方案耗时vs 串行瓶颈
串行993ms1.00×—
ProcessPool（论文）~4300ms4.33×pickle→IPC→tokenize→IPC←unpickle
ThreadPool（本方案）778ms0.78×Rust 释放 GIL → 共享内存零拷贝

关键洞察： Rust tokenizer 释放 GIL → ThreadPool 单进程安全并行，避开 IPC 成本。单核上比 ProcessPool 快 4-5×。

05 · vLLM 集成分析

Chat Completion └── render_messages() → apply_chat_template(..., tokenize=True) └── tokenizer.encode(text) ← 单线程串行 → LoPT 插入点：替换 encode() 为并行版本

注意： AsyncMicrobatchTokenizer 只做跨请求批处理（2ms窗口合并），不加速单请求。

场景CPULoPT
GPU 推理为主低高（TTFT↓）
高并发高低
Prefix Cache 命中—无
离线 batch中中高

Tokenize 在 vLLM 中占比不高（GPU prefill 为主）。LoPT 降低 TTFT，对吞吐帮助有限。

06 · 多核扩展

ThreadPool 受 GIL 限制——仅 Rust tokenize 能并行。16 核机器也只能用 1-2 核。

方案单核4核16核成本
① ThreadPool1.35×~2.0×~2.3×低
② ProcessPool0.7×~1.8×~3.5×中
③ mmap 共享内存0.7×~2.0×~4.2×高
④ Rust rayon2.0×~7.0×~10×+很高
  1. 当前： ThreadPool, 4-8 核, 1.3-2.0×
  2. 中期： mmap 共享内存, 突破 GIL, 2-4×
  3. 长期： fork tokenizers + rayon, 5-10×
